## 一、确认成立（可直接照做）

以下几条对照 repo 代码和 AWS 官方文档，完全站得住：

### ✅ 1.1 缺 `provision_permission_set` — **必改**
- AWS 官方 API 文档明确写：`AttachManagedPolicyToPermissionSet` / `PutInlinePolicyToPermissionSet` / `AttachCustomerManagedPolicyReferenceToPermissionSet` 之后，*"If the permission set is already referenced by one or more account assignments, you must call `ProvisionPermissionSet` to apply the corresponding IAM policy updates to all assigned accounts."*
- `scripts/restore_permission_sets.py` 的 update 分支确实没有 provision 调用 — 首次 clone 场景下 target 还没 assignment 不会暴露，但 RUNBOOK §7.3 回滚重跑场景**会静默不生效**。
- review 的修复代码片段（`provision_permission_set` + `describe_permission_set_provisioning_status` 轮询 SUCCEEDED）是标准做法。

### ✅ 1.2 Applications 本身没备份 — **必改**
- 读了 `upstream/mist/backup.py`：它只调 `list_application_assignments`，确实不导出 application 定义、SCIM endpoint、SAML metadata、TTI 配置。
- AWS SCIM endpoint + access token 在 target 实例是**全新生成**的（AWS 文档：SCIM token 一年有效期，新实例就是新 token），Kiro/Okta 等下游必须重新粘贴新 endpoint/token，没有 checklist 客户会漏。
- review 建议的 `backup_applications.py` 或至少 manual checklist，是 DR 场景的硬需求。

### ✅ 1.3 跨 Org 缺 account_id 映射 — **必改**
- 读了 `upstream/mist/restore.py`：`create_account_assignment` 直接用了 backup 里原样的 account id 做 `TargetId`，跨 Org 时 source 的 account id 在 target Org 里不存在 → 100% 失败。
- RUNBOOK §0 自己就说 "DR 推荐新 Org"，但 §9 把 account_id_map 列成 TODO — 逻辑自相矛盾。review 指出这是"当前方案不支持跨 Org"是准确的。

### ✅ 1.4 异常捕获过宽 — **强烈建议改**
- `backup_permission_sets.py` 里 `dump_permission_set()` 三处裸 `except Exception:` / `except:`（inline policy、boundary、tags）全部把 AccessDenied 和 ResourceNotFound 混在一起。
- `restore_permission_sets.py` `delete_inline_policy_from_permission_set` 和 `delete_permissions_boundary_from_permission_set` 用 bare `except ClientError: pass` — 真的权限问题会被吞成"本来就没东西"。
- 修复建议（只吞 `ResourceNotFoundException`）是正确做法。

### ✅ 1.5 `backoff` 声明了却没用 — **属实**

### ✅ 1.7 tag 不是 reconcile — **属实**

### ✅ 2.1 密码/MFA 不可迁移 — **属实且严重**

## 二、需要调整（3 条）

### ⚠️ 2.1 事实偏差：delegated admin 不能解决"管理账号被删"

review §1 把"Org 管理账号被删除/suspend"列为账号级故障触发场景，然后推荐 "delegated administrator + external IdP" 作为正解。

**问题：** delegated admin 只是把 Identity Center 的**日常管理**委托给另一个成员账号，但 **Identity Center 实例本身仍然绑定在管理账号下**（AWS 官方文档：*"Only a user or role in the Organizations management account can configure a member account as a delegated administrator"*，且 Identity Center 的服务-linked role `AWSServiceRoleForSSO` 创建在管理账号里）。

**结论：** 
- 管理账号 suspend / 合规冻结 → delegated admin **不能救**（因为整个 Org 挂了，Identity Center 实例跟着挂）
- 管理账号 root 被锁 → delegated admin **可以救**（成员账号还能操作）
- 管理账号被误删 → delegated admin **不能救**

所以 review §1 把 delegated admin 当成"账号级故障"的通用解是不准确的。真正的跨 Org DR 仍然需要 clone 或者 external IdP。external IdP 部分 review 说得对（确实能让 Identity Center 降级成 SAML consumer），但 delegated admin 不在这条链路上。

**建议 review 作者把 §1.1 改成：**
> "Identity Center 管理委托给独立 identity 账号可以减少日常误操作暴露面，但**不能解决管理账号本身故障的场景**。真正的跨 Org DR 路径是：external IdP（首选）+ 本 repo 的 permission set/assignment 克隆（次选）"

### ⚠️ 2.2 优先级建议：§2.4 RPO 24h 应该升级到 P0

review 把"日备份 RPO=24h"列在 P2「运维可行性」里，但自己的论证（"新员工登不进、已离职员工仍有权限，合规红线"）已经把问题性质说清楚了 — **这是合规问题不是运维问题**。

对于 1000+ 用户规模、涉及 Kiro 订阅的客户，"故障前一天内离职员工仍有权限"可能直接违反 SOX / 等保 / GDPR。**建议提到 P0/P1**。缓解成本很低（CloudTrail → EventBridge → Lambda 增量），作者也写了"成本极低"，没理由延后。

### ⚠️ 2.3 措辞建议：§1.6 手写 paginate 不是必须要改

review 说手写 `paginate()` 函数"在 NextToken 非常大或 API 有 session 语义时容易漏页"。

**实际情况：** 
- 读了 `scripts/backup_permission_sets.py` 和 `restore_permission_sets.py` 的 `paginate()` 实现，是标准的 `while "NextToken" in r` 模式，和 boto3 paginator 行为等价，**没有逻辑 bug**。
- boto3 paginator 的好处主要是 `PaginationConfig`（PageSize / MaxItems / StartingToken），以及 `search()` JMESPath — 但本项目这几个脚本都不需要。
- 改成 `client.get_paginator(...)` 是**优雅性提升**而不是**正确性修复**。

**建议：** 这条维持 P1 但措辞降调，别让客户以为现在的 paginate 有 bug。

## 三、review 没提到的点（建议补进去）

### 🆕 3.1 `mist/restore.py` 里 `username.lower()` 的隐患
读了 `upstream/mist/restore.py`：

```python
return {sub['UserName'].lower():sub['UserId'] for sub in results}
```

它把 UserName 全部 lowercase 做 key。AWS Identity Center 的 UserName 本身是**大小写不敏感**的（SCIM RFC 7643 §4.1.1 规定），但是：
- 如果 source 有 `Alice@corp.com` 和 `alice@corp.com` 同时存在（理论上 Identity Center 会拒绝，但旧数据可能有）— 会静默丢一个
- 如果 source UserName 是 `Alice@Corp.com`，backup 里原样保存，restore 时 mist 用 lowercase 去 target 查找 — 如果 target 因为任何原因 case 不一致，会找不到

review 没提这个，但 1000 用户规模下至少该有个 "UserName 大小写一致性校验" 的前置检查。不影响主要方向，P2 补一下。

### 🆕 3.2 SCIM token 一年过期是个计时炸弹
AWS SCIM 文档明确：SCIM access token 一年有效期，过期后同步停止（AWS 只在 90 天内提醒）。

如果客户走 review 推荐的 "external IdP + target Identity Center SCIM 同步" 路径，这个 token 是在 target 实例里，**平时不用所以没人 rotate**。灾难发生日如果 token 正好过期 → 即便 target 实例活着，用户/组也不会被同步进去。

**建议：** 加到 RUNBOOK §7.1 日常维护里，要求 target 侧 SCIM token 和 source 同周期 rotate，并且做 CloudWatch alarm（`IdentityCenter SCIM Token Expiring` 是 AWS 发出的 EventBridge event）。review 没提，但这是"平时不练、战时必翻车"的典型坑。
