# identity-center-backup — 架构 & 代码 Review

> 评审人：首席架构师 🏛️（AWS PSA × 客户 PA 双视角）
> 日期：2026-04-23
> 范围：`README.md` / `SKILL.md` / `scripts/*.py` / `docs/RUNBOOK.md` / `upstream/mist` 依赖行为
> 整体判断：**方向正确、脚本可读性不错，但距离"可在生产落地的 DR 方案"还差一截**。不建议直接给客户执行，先补齐本文 🟥 级问题。

---

## 0. 一句话结论

脚本写得比 `upstream/mist` 干净（paginate 封装、dry-run、idempotent 都到位），但整个**方案定位**存在根本性问题：DR 需求的正解是 *delegated admin + external IdP + IaC*，而本 repo 把"账号级克隆"做成主路径、架构修正做成附注。再加上 provision、application、account-id 映射三个硬缺口，目前拿去做 1000+ 用户的 Kiro 灾备**会在切换当天踩坑**。

## 1. 【AWS PSA 视角】

### 🟥 P0 — 必须修

#### 1.1 `restore_permission_sets.py` 缺 `provision_permission_set` 调用
update_permission_set / 策略 reconcile 完成后，AWS 要求显式调用 `ProvisionPermissionSet` 才能把变更推送到已有 assignment 上。首次迁移 target 无活动 assignment，确实不会立即暴露问题；但 RUNBOOK §7.3「回滚→重跑 restore」场景下，target 已经有 assignment，不 provision 就是**静默不生效**。

*修：* 在 reconcile 的最后加：
```python
sso.provision_permission_set(
    InstanceArn=instance_arn, PermissionSetArn=arn,
    TargetType="ALL_PROVISIONED_ACCOUNTS",
)
# 然后 describe_permission_set_provisioning_status 轮询到 SUCCEEDED
```
只在"存在 assignment"时才 provision，避免无意义 API 调用。

#### 1.2 没有 Applications 本身的 backup
`upstream/mist` 只备份 **application assignments**，不备份 application 定义（SAML 配置、SCIM endpoint、ACS URL、Trusted Token Issuer）。RUNBOOK §8 里轻描淡写一句"target 侧先重建 applications"，实际上：
- Kiro/Okta 这类 SCIM application 的 endpoint/token 在 target 是全新的 → 客户侧应用必须重接
- 没有任何脚本帮客户导出 current application 清单做 checklist

*修：* 加 `backup_applications.py`（至少导出 `ListApplications` + `DescribeApplication` + `ListApplicationAssignmentConfigurations`），配 `restore_applications.py` 或至少输出一份"手动重建 checklist"。

#### 1.3 缺 account_id 映射（跨 Org 场景直接崩）
RUNBOOK §0 明确说 DR 场景推荐"新 Org"，但 `mist/restore.py` 里 assignment 的 `TargetId` 是 source 的 account id，到新 Org 一个都对不上。`docs/RUNBOOK.md` §9 把这条列成 TODO — **意味着当前版本不支持跨 Org DR**，与场景需求冲突。

*修：* 在 docs/RUNBOOK.md §0 决策树里**明确标注**："同 Org DR → 本方案可用；跨 Org DR → 必须先做 `account_id_map.json`，否则不能用。" 并把 map 脚本补上（优先级比其它 TODO 高）。

### 🟧 P1 — 强烈建议修

#### 1.4 异常捕获过宽，吞真错
多处 `except Exception:` / `except ClientError:` 把"没权限"和"真的没配置 inline policy"混在一起：
- `backup_permission_sets.py` 的 `get_inline_policy_for_permission_set`、`get_permissions_boundary_for_permission_set`、`list_tags_for_resource` 全部裸 except
- `restore_permission_sets.py` `delete_inline_policy` / `delete_permissions_boundary` 用 bare `except ClientError` 忽略 — 真的 AccessDenied 会变成"没东西要删"，数据静默错误

*修：* 判断 `ResourceNotFoundException` 才忽略，其余要 raise 或至少 log.error。

#### 1.5 `backoff` 在 requirements.txt 里但代码没用
依赖声明了 `backoff>=2.2.1`，代码里零引用。1000+ 用户 CreateUser 靠 boto3 default retry（standard, 3 次）在实测中遇 throttle 会失败。要么 **真的用上** `@backoff.on_exception(backoff.expo, ClientError, max_tries=10)`，要么删依赖。

#### 1.6 `paginate()` 不是真 paginator
手写翻页在 NextToken 非常大或 API 有 session 语义时容易漏页，而且错失 boto3 内置的 `PaginationConfig` / retry 集成。

*修：*
```python
paginator = client.get_paginator(method_name)
for page in paginator.paginate(**kwargs):
    yield from page.get(result_key, [])
```

#### 1.7 `restore_permission_sets.py` tag 不是 reconcile
create 时传了 `Tags`，update 路径**完全没碰 tag** — 备份说是"全字段同步"，实际 tag 只进不出/只初始化不更新。

*修：* update 路径补 `TagResource` / `UntagResource` 的 diff 应用。

#### 1.8 IAM 权限清单格式混乱
RUNBOOK §3.1 / §3.2 里的 IAM 权限写成逗号+换行+裸 action 名混排，且 `sso:*` / `identitystore:*` 前缀不统一（有的带有的不带）。直接给客户贴到 IAM policy 会语法错误。

*修：* 给一个真实可用的 `docs/iam-policy-source.json` + `iam-policy-target.json`，runbook 里 reference 过去，不在正文贴散装 action 列表。

### 🟨 P2 — 可以之后补

- 没有 S3 archive + Object Lock + CRR 的 IaC（runbook §4.4 只是一段话）
- 没有测试：paginate、`build_create_user_payload`、memberships 翻译这几个都是纯函数，适合 pytest
- `import time; time.sleep(0.2)` 硬编码节流，应改成参数
- `default=str` 兜底 JSON 序列化会把 datetime 变字符串但不提示 — 正式产物里应该显式 isoformat

## 2. 【客户 PA 视角】

### 🟥 方案根本性担忧

#### 2.1 密码 + MFA 不可迁移 = 切换当天的组织级事件
- 1000 Kiro 用户**同时收邀请邮件**、**同时重设 MFA**，IT helpdesk 当天瘫痪是确定事件
- 这不是"无缝切换"，这是"强制全员重注册"
- RUNBOOK §8 把它列成"已知局限"，但业务需求里"主账号故障时可无缝切换"跟这个硬伤**直接冲突**

**作为架构决策者，我需要：** 要么客户接受"切换日有 0.5–1 天 IT 中断"（写进 RTO 假设），要么**强烈推动改用 external IdP**（Entra / Okta / Google），把 Identity Center 降级成 SAML consumer。后者才是真正的 DR 架构。

#### 2.2 切换流程太手工
RUNBOOK §7.2「通知用户切换到新 start URL」—— 1000 人规模下这不是切换，是一次项目。生产 DR 必须有：
- DNS 层封装（`login.example.com` CNAME → 当前活跃 start URL）
- Kiro 等下游应用的 endpoint 抽象
- 切换演练剧本（季度）

当前方案缺这个封装层 → 切换 RTO 无法保证。

#### 2.3 没有实测 → 不能用于生产
RUNBOOK §9 TODO 第 1 条："实测一遍端到端（需要一对测试账号）"。**未经端到端验证的 DR 方案等于没有 DR**。客户接手前必须先：
- 跑一对沙箱账号完整走通 backup → restore → 登录
- 记录实际 RTO（从故障宣告到最后一个用户登录成功）
- 记录 Customer Managed Policy 预铺工作量

#### 2.4 日备份 RPO = 24h，对用户目录偏长
Identity Center 里用户入/离职、权限变更每天都在发生。RPO 24h 意味着**故障前一天内的人员变动全丢**（新员工登不进、已离职员工仍有权限）。这是合规红线。

*建议：* 用户 CUD 事件走 CloudTrail → EventBridge → 增量备份（或至少小时级）。成本极低。

### 🟧 运维可行性

#### 2.5 Customer Managed Policy 预铺成本被低估
RUNBOOK §5.4 就一句「前提是同名 IAM policy 必须已经存在于所有被 assign 的成员账号里」。实际上这意味着：
- 所有 CMP 在 target Org 每个成员账号都要用 IaC 铺一遍
- 数量 = permission_sets × 成员账号，成百上千不稀奇
- **谁来维护两套 Org 的 CMP 同步？** 不解决这个，备份脚本跑完也用不了

*建议：* 把 CMP 铺设脚本（StackSets）列为前置条件的 **第一阶段**，并在 `backup_permission_sets.py` 输出时生成一份 `CustomerManagedPolicyManifest.json`，列出所有需要预铺的 `(account_id, policy_name)` 对，给 StackSet 用。

#### 2.6 成本/时间估算缺失
客户决策需要的关键数字 runbook 全都没写：
- Restore 1000 用户总耗时？（估算 2–5 分钟，但没实测数据）
- Target 账号开通 + IAM Identity Center 启用 + 企业支持审核？（通常 1–3 天）
- CMP 预铺工作量（人日）？
- 日备份 + S3 存储 + Lambda 费用？（量级应该很小，但要给数字）

### 🟨 文档细节

#### 2.7 README 中文版的分工图 ASCII 画错了
```
│ Assignments ─────────┼──► .json ┼──► upstream/mist/    │
│ App Assignments      │          │         restore.py   │
```
Users/Groups 那几行和 Assignments 行的箭头出入方向混乱，影响阅读。

#### 2.8 `RUNBOOK.md` §6.1 验证脚本嵌在 heredoc 里
直接在 runbook 里贴 `python3 <<'PY'` 不如做成 `scripts/verify_counts.py`，方便重复使用 + 写测试。

## 7. 🟥 Kiro 专项补充（2026-04-23 第二轮，读完 <https://kiro.dev/docs/enterprise/iam/> 后）

**前情：** 第一轮 review 把 Kiro 当成普通 SCIM 下游应用看，现在读完 Kiro IAM 文档发现本方案**根本不覆盖 Kiro 订阅数据面**。这一条直接改写结论。

### 7.1 Kiro 订阅的真实数据模型（文档关键事实）

Kiro enterprise 在 AWS 账号里实际落地的东西有四层：

| 层 | 存储位置 | 本 repo 覆盖？ |
|----|---------|--------------|
| IAM Identity Center users / groups / permission sets / assignments | `identitystore` + `sso-admin` | ✅ 覆盖（见第一轮 P0） |
| **Kiro Application**（IdC 里的 app 实例，带 TIP、Auth Method、Grant、Assignment Configuration） | `sso-admin` + `signin` (TIP) | 🟥 **完全不覆盖** |
| **Kiro 订阅本身** (user-subscriptions claim) | `user-subscriptions:Claim`（每 user/group 一条 claim） | 🟥 **完全不覆盖** |
| Kiro 侧用户资产（profile、历史、CodeWhisperer/Q Developer tagging） | Kiro 后端 + CodeWhisperer + Q Developer，以 IdC UserId 为 key | 🟥 **完全不覆盖**（且 UserId 会在 target 重建后变化） |

文档里出现的关键 API / SLR：
- `sso:CreateApplication` / `PutApplicationAuthenticationMethod` / `PutApplicationGrant` / `PutApplicationAssignmentConfiguration`
- `signin:CreateTrustedIdentityPropagationApplicationForConsole` — **Kiro 用 TIP 把身份往下游（Q Developer / CodeWhisperer / CodeGuru Security）传**
- `user-subscriptions:CreateClaim / ListClaims / SetOverageConfig` — **订阅本身是 claim 记录，不是 IdC 对象**
- SLR：`AWSServiceRoleForUserSubscriptions` + `AWSServiceRoleForAmazonQDeveloper`（订阅时自动创建）

### 7.2 对 DR 方案意味着什么（🟥 严重）

1. **Kiro Application 必须走 Kiro console 在 target 重建。** 不是普通 SAML app —— 带 TIP、grant、authentication method、assignment configuration。`upstream/mist` 只备份 assignment 不备份 application 本体，用 `sso-admin` 手写复原几乎不可能一比一还原。*正路子*：target 账号走 Kiro console → 订阅 → 让 Kiro 自己调 `sso:CreateApplication` 创建完整一套。
2. **1000+ 订阅 claim 必须在 target 重新发放。** `user-subscriptions:CreateClaim` 绑定的是 IdC 的 **UserId / GroupId**；restore 脚本只保 UserName/DisplayName 一致，UserId 是新账号重新分配的。⇒ 切换当天要在 target Kiro console 把 1000 人**重新 subscribe 一遍**。这不是「恢复」，是「重新开通」。
3. **Trusted Identity Propagation 链路断裂。** Kiro 通过 TIP 把身份往 Q Developer / CodeWhisperer 下游传，target 账号的 TIP application 全新 → 用户在 Kiro 里的历史 profile、CodeWhisperer tagging、Q Developer dashboard 数据**大概率不跟随过来**（按 claim binding 推断是丢的；需找 Kiro 团队确认）。
4. **成本触点被忽略。** target 账号要**独立购买一份 Kiro 企业订阅**（双份费用，直到切换完成）。RUNBOOK 里零字提及 Kiro 侧商务动作。1000 seat 的双份订阅成本很可观，客户决策必须看到这个数字。

### 7.3 结论升级：原始需求无法通过本方案达成

> 客户需求：「1000+ Kiro 订阅用户通过 Identity Center 登录主账号，需要在另一个备用账号克隆一套 Identity Center，主账号故障时可无缝切换。」

- 本 repo 能做到：AWS SSO 登录能力复原（users/groups/PS/assignment）
- 本 repo 做不到：Kiro application、Kiro 订阅 claim、Kiro 侧用户资产 —— **恰好是「Kiro 订阅用户」这句话的核心**

⇒ 切换当天实际发生的事：
1. 用户切到 target start URL 登录 AWS ✅（靠本 repo）
2. 用户打开 Kiro 发现**没订阅** ❌
3. 管理员在 target Kiro console 重新 subscribe 1000 人 🟥（1–2 天工作量）
4. 用户历史 conversation / profile 可能丢失（需 Kiro 团队确认）

**这叫「冷备」不叫「无缝切换」。** 真实 RTO 至少以天计，和客户需求差一个量级。

### 7.4 必须加进 repo 的内容（新 P0，优先级高于第一轮 P0）

#### 7.4.1 RUNBOOK 必须诚实披露覆盖边界
在 §0 决策树后加一节：
```
本方案覆盖：AWS IAM Identity Center 层（登录能力）
本方案不覆盖：Kiro Application、Kiro 订阅 claim、Kiro 侧用户资产
  ⇒ 切换当天需要在 target Kiro console 独立完成：
     - 开通 Kiro 企业订阅（新账单）
     - 重新创建 Kiro Application（Kiro console 自动做）
     - 为 1000+ 用户/组重新发放 subscription claim
     - 和 Kiro 团队对齐用户历史数据迁移（大概率不可迁移）
  ⇒ 真实 RTO 下限：> 1 天；订阅侧 RPO = 100%（无增量备份可能）
```

#### 7.4.2 补两个脚本
- `scripts/backup_kiro_subscriptions.py` — 调 `user-subscriptions:ListClaims` / `ListApplicationClaims` / `ListUserSubscriptions`，导出 claim 清单（记录哪些 UserName/GroupName 有订阅 + `SetOverageConfig` 设置）。
- `scripts/restore_kiro_subscriptions.py` — 在 target 已建好 Kiro Application 后，按 UserName/GroupName 在新 IdC 反查 UserId/GroupId，调 `user-subscriptions:CreateClaim` 批量重发。依赖：target 账号已手动完成 Kiro 订阅开通。
- 权限：`user-subscriptions:*` + `sso:ListApplications` + `identitystore:ListUsers/ListGroups`。

#### 7.4.3 Kiro Application 重建走 console 而非 API
RUNBOOK 明确写：**不要尝试用 `sso-admin create-application` 手工复原 Kiro application，必须走 Kiro console 订阅流程自动创建**。原因：TIP + authentication method + grant 的组合 Kiro 会随版本调整，手工复原会在下次 Kiro 功能升级时踩坑。

#### 7.4.4 Overage config 必须先于切换
`SetOverageConfig`（seat 超额计费模式）是每个订阅的独立配置，必须单独备份 + 在切换前设置到和 source 一致，避免切换当天客户突然看到账单模式变了。

### 7.5 强烈建议：把项目分层重新定位

当前一个 repo 名字（identity-center-backup）承诺解决 Kiro DR，会误导客户。建议分三层：

- **Layer 1 — IdC 层 DR**（即当前 repo），明确边界是「登录能力」
- **Layer 2 — Kiro 订阅层 DR**（新增），脚本 + 和 Kiro 团队对齐的 runbook
- **Layer 3 — 应用数据层**（Kiro conversation、CodeWhisperer profile） —— 这一层**必须靠 Kiro 产品路线图解决**，自建方案做不到；写清楚「产品依赖项」

客户看到这三层才能判断「无缝切换」在当前产品形态下是否可行。只给 Layer 1 客户会误以为问题解决了，切换当天出事就是信任事故。
