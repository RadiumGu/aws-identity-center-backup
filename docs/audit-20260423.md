## TL;DR

The sprint landed 6 of 8 improvements correctly (B1 provision, C1 narrowed
exceptions, C3 tag reconcile, C7 SCIM ops, C8 UserName case check, IAM JSON
policies). But the Kiro subscription work (B3) was rewritten under the wrong
assumption that `user-subscriptions` is a public boto3 service — it is not.
This caused both new scripts to crash on import. I patched the SigV4 wrapper
back in and fixed two reader scripts that were also stale against the new
JSON schema. See §2 for details.

## 2. Regressions I had to fix 🔴

### 2.1 `boto3.client("user-subscriptions")` is a crash

Both `backup_kiro_subscriptions.py` and `restore_kiro_subscriptions.py`
instantiated `boto3.client("user-subscriptions")`. boto3 1.42.x has no such
service — `UnknownService` at import of the client. The improvement-plan
implicitly assumed this service had been promoted out of private status; it
has not. Evidence:

- classmethod reverse-engineering (Feb 2026) — the service is still on the
  private `service.user-subscriptions.<region>.amazonaws.com` Coral endpoint
  and not exposed in CLI/SDK.
- Local verification: `boto3.Session().get_available_services()` does not
  include `user-subscriptions`.

**Fix landed**:
- New `scripts/_user_subscriptions.py` — a SigV4-signed shim that exposes a
  boto3-client-shaped surface (`list_user_subscriptions`, `list_claims`,
  `list_application_claims`, `create_claim`, `describe_overage_config`,
  `set_overage_config`) and converts HTTP errors to `ClientError`.
- `backup_kiro_subscriptions.py` / `restore_kiro_subscriptions.py` now use
  `UserSubscriptionsClient()` instead of `boto3.client`.
- `restore_kiro_subscriptions.py` prints a LOUD banner on start reminding
  the operator that `CreateClaim` / `SetOverageConfig` have historically
  returned HTTP 500 on external calls and to fall back to the manual Kiro
  console flow if they do.

### 2.2 Schema drift in reader scripts

`scripts/kiro_restore_checklist.py` and `scripts/check_kiro_target_readiness.py`
still read `data["Subscriptions"]` — the **old** schema from the original
`backup_kiro_subscriptions.py`. The sprint rewrote backup to emit
`UserClaims` / `GroupClaims` but did not update the two consumers. This
silently breaks both scripts against any new backup.

**Fix landed**: both scripts now accept both schemas (new `UserClaims`/
`GroupClaims` and legacy `Subscriptions`). No behavior change for existing
snapshots.

## 3. Still-open items / worth watching ⚠️

### 3.1 IAM `user-subscriptions:*` actions may not be enforced

`docs/iam/iam-policy-target.json` grants `user-subscriptions:*`, but the
service name and action verbs are private. Two things might happen:

- IAM accepts the policy but silently no-ops (since the service isn't in
  the authorizer's registry), and calls are authorized via other means
  (e.g., sso-admin permissions + service-linked role).
- IAM rejects the policy on `aws iam create-policy` with a syntax error.

I have not been able to verify either without a live account. Suggested
action: at first sandbox test, try attaching the policy and see. If IAM
rejects, remove the `user-subscriptions:*` block and rely on the Kiro
service-linked roles.

### 3.2 CreateClaim pathway is still speculative

Even with the SigV4 plumbing correct, the `CreateClaim` verb might not be
the one AWS actually uses internally (it's a guess based on `ListClaims`
symmetry). If all `CreateClaim` calls return 500, that's the expected
path — the manual Kiro console remains the source of truth. The
`try_kiro_create_assignment.py` probe is still useful to re-run quarterly
to detect the moment AWS un-gates the API.

### 3.3 B2 (account_id map) still missing

Sprint summary lists this as blocked on format confirmation. In the
meantime, `upstream/mist/restore.py` will silently fail on TargetId
mismatch in cross-Org scenarios. Minimum viable version: add a pre-flight
check that compares Source and Target account IDs and aborts with a clear
message if they differ, deferring the full mapping table to a later
sprint.

### 3.4 B5 end-to-end test still blocks release

Correctly called out in the sprint summary. Release gate is correct;
nothing to add.

### 3.5 Minor polish worth batching later

- `backup_kiro_subscriptions.py` still uses `paginate()` from local scope
  for `list_applications` — fine, but inconsistent with the rest of the
  file which also imports `paginate_private`. Cosmetic.
- `_user_subscriptions.py` returns response keys inspecting both
  PascalCase and camelCase; once we see a real response, we can trim to
  the actual casing.
